# PendingApproval Provisioner Support — Design Spec

**Issue:** casehubio/casehub-ops#13
**Date:** 2026-06-30
**Status:** Draft

## Problem

The desiredstate runtime (casehub-desiredstate#14, shipped) provides a complete approval lifecycle for provisioning operations. Two orthogonal mechanisms exist:

1. **`requiresHuman` + `HumanNodeHandler`** — compile-time, structural. For nodes that cannot be automated. Covered by ops#28 (separate issue).
2. **`PendingApproval` + `PendingApprovalHandler`** — runtime, dynamic. The provisioner evaluates risk and decides whether to request human sign-off before proceeding.

This spec covers mechanism 2. All four ops provisioners currently hardcode `requiresHuman=false` and never return `PendingApproval`. The runtime's `NoOpPendingApprovalHandler` fails when `PendingApproval` is returned. The infra domain has existing plan/apply infrastructure (`ProvisionPhase`, `RiskThresholds`, `InfraBackend.plan()`) designed for this flow but never wired up.

## Runtime Architecture (as shipped)

The `SimpleTransitionExecutor` orchestrates the approval lifecycle:

```
For each node in the transition plan:

  1. pendingApprovalHandler.check(node, action, tenancyId)
     → None:     fall through to provisioner
     → Pending:  return Skipped (node stays unprovisioned, no fault)
     → Approved: enrich context with approval, fall through to provisioner
     → Rejected: acknowledgeRejection(), return Rejected (fault feedback fires)

  2. provisioner.provision(node, context)
     → Success:         return Succeeded
     → Failed:          return Failed (fault feedback fires)
     → PendingApproval: handler.recordPending(), return Skipped
```

Key types in `desiredstate-api`:

| Type | Purpose |
|------|---------|
| `ProvisionResult.PendingApproval(nodeId, planReference)` | Provisioner signals approval needed |
| `PlanApproval(planReference, approvedBy, approvedAt)` | Carried in context after approval |
| `ProvisionContext.hasApproval()` / `.withApproval()` | Re-entry detection |
| `PendingApprovalHandler` | SPI: check, recordPending, acknowledgeRejection |
| `ApprovalCheckResult` | Sealed: None, Pending, Approved, Rejected |

## Design

### 1. Shared approval types — `io.casehub.ops.api.approval`

New package in the `api` module containing shared types used by all domain provisioners.

#### `RiskClassification`

Moved from `io.casehub.ops.api.infra.context`. No wrapper, no re-export — callers update imports.

```java
public enum RiskClassification { LOW, MEDIUM, HIGH, CRITICAL }
```

#### `ApprovalThresholds`

Replaces `RiskThresholds` from infra. Generalised name, same semantics.

```java
public record ApprovalThresholds(
    RiskClassification autoApproveBelow,
    boolean requireSecondReviewer
) {
    public boolean requiresApproval(RiskClassification risk) {
        return risk.compareTo(autoApproveBelow) >= 0;
    }
}
```

Semantics: `requiresApproval(HIGH)` with `autoApproveBelow=HIGH` returns `true` — "at or above the threshold."

#### `PlanDetail`

Marker interface for domain-specific plan data.

```java
public interface PlanDetail {}
```

Infra's `ProvisionPlan` gains `implements PlanDetail`. Other domains create their own implementations as needed.

#### `ApprovalPlan`

The plan record stored for human review and re-entry validation.

```java
public record ApprovalPlan(
    NodeId nodeId,
    StepAction action,
    RiskClassification risk,
    String summary,
    String tenancyId,
    int specHash,
    PlanDetail detail
) {}
```

- `summary`: human-readable description generated by the domain evaluator
- `specHash`: `node.spec().hashCode()` at plan generation time — used for stale-approval detection on re-entry
- `detail`: domain-specific structured plan data (nullable). Infra stores `ProvisionPlan`; deployment may store nothing

#### `ApprovalDecision`

Result of evaluating whether a node needs approval.

```java
public sealed interface ApprovalDecision {
    record AutoApproved() implements ApprovalDecision {}
    record RequiresApproval(ApprovalPlan plan) implements ApprovalDecision {}
}
```

`RequiresApproval` carries the full `ApprovalPlan` — risk classification and plan generation happen in a single pass.

#### `ApprovalEvaluator`

Interface for domain-specific risk evaluation and plan generation.

```java
public interface ApprovalEvaluator {
    ApprovalDecision evaluate(DesiredNode node, StepAction action, String tenancyId);
}
```

Each domain provides an `@ApplicationScoped` implementation.

#### `PlanStore`

Stores approval plans by reference ID.

```java
public interface PlanStore {
    String store(ApprovalPlan plan);
    Optional<ApprovalPlan> retrieve(String planReference);
    void remove(String planReference);
}
```

In-memory `@ApplicationScoped` implementation (`InMemoryPlanStore`) using `ConcurrentHashMap<String, ApprovalPlan>`. Generates UUID strings as plan references.

### 2. OpsPendingApprovalHandler

`@ApplicationScoped` in `io.casehub.ops.api.approval`. Displaces the runtime's `NoOpPendingApprovalHandler` (which is `@DefaultBean`).

#### State model

Each pending approval is keyed by `(nodeId, action, tenancyId)`:

```
PENDING  →  APPROVED  (via approve())
PENDING  →  REJECTED  (via reject())
REJECTED →  removed   (via acknowledgeRejection())
```

#### SPI methods (called by SimpleTransitionExecutor)

- **`check()`**: Returns the stored state. `Approved` is NOT consumed on read — persists so failed provisions can retry on next reconciliation cycle. Stale approvals are harmless (provisioned nodes leave the plan) or detected by the provisioner (specHash mismatch).
- **`recordPending()`**: Stores/replaces entry as PENDING. Returns `StepOutcome.Skipped`. Naturally supersedes any stale approval for the same key.
- **`acknowledgeRejection()`**: Removes the entry entirely.

#### External API (beyond the SPI)

- `approve(nodeId, action, tenancyId, approvedBy)` — transitions PENDING → APPROVED, creates `PlanApproval`
- `reject(nodeId, action, tenancyId, reason)` — transitions PENDING → REJECTED

These methods are the integration seam for WorkItem completion handlers, REST endpoints, or test code.

#### Implementation details

- In-memory `ConcurrentHashMap`. Bounded: at most one entry per (nodeId, action, tenancyId).
- Reconciliation after approval relies on ReconciliationLoop's periodic resync. Responsive `requestReconciliation()` is a follow-up.
- No casehub-work dependency. WorkItem integration is a layer on top.

### 3. Provisioner flow

Every ops provisioner follows the same pattern:

#### First call (no approval in context)

```
provision(node, context):
  if context.hasApproval() → re-entry flow

  decision = evaluator.evaluate(node, PROVISION, context.tenancyId())

  if AutoApproved → doProvision(spec, context), return Success/Failed

  if RequiresApproval(plan):
    planRef = planStore.store(plan)
    return PendingApproval(node.id(), planRef)
```

#### Re-entry (after approval)

```
provision(node, context):
  approval = context.approval()
  storedPlan = planStore.retrieve(approval.planReference())

  if storedPlan absent → strip approval, re-evaluate from scratch
  if storedPlan.specHash() != node.spec().hashCode() → spec changed, re-evaluate
  otherwise → doProvision(spec, context), return Success/Failed
```

The stale-approval guard prevents an old approval from authorising a different change. If the spec changed between plan generation and re-entry, the provisioner re-evaluates risk and may issue a new `PendingApproval`.

#### Deprovision

Same pattern through `deprovision()` / `DeprovisionContext` / `DeprovisionResult.PendingApproval`.

### 4. Domain-specific changes

#### Deployment domain

**`DeploymentApprovalEvaluator`** classifies risk by spec type:

| Spec type | Risk | Rationale |
|-----------|------|-----------|
| `TrustPolicyNodeSpec` | HIGH | Affects routing based on trust scores |
| `AgentNodeSpec` | MEDIUM | Agent capability changes |
| `ChannelNodeSpec` | LOW | Communication channel setup |
| `CaseTypeNodeSpec` | LOW | Case definition registration |
| `EndpointNodeSpec` | LOW | API endpoint registration |

Default thresholds: `autoApproveBelow = HIGH` — only trust policy changes require approval.

Summary generation produces descriptions like: `"Provision trust policy 'claims-routing': threshold=0.85, blendFactor=0.3"`.

**`DeploymentNodeProvisioner` changes:**
- Inject `ApprovalEvaluator` and `PlanStore`
- First call: evaluate → if `RequiresApproval` → store plan → return `PendingApproval`
- Re-entry: retrieve plan → validate specHash → proceed to `doProvision()`
- Existing handler dispatch moves into private `doProvision()` — handlers themselves do not change
- Same pattern for `deprovision()`

#### Infra domain

**`InfraApprovalEvaluator`** delegates to `InfraBackend.plan()`:

1. Call `backend.plan(spec, planContext)` → `Optional<ProvisionPlan>`
2. Empty → `AutoApproved` (no changes needed)
3. Present → check `provisionPlan.risk()` against thresholds
4. Below threshold → `AutoApproved`
5. Above → `RequiresApproval(approvalPlan)` with `ProvisionPlan` as `PlanDetail`

This wires up the infra domain's existing plan/apply infrastructure.

**`InfraNodeProvisioner` changes:**
- Inject `ApprovalEvaluator` and `PlanStore`
- First call: evaluate → if `RequiresApproval` → store plan → return `PendingApproval`
- Re-entry: retrieve `ProvisionPlan` from `ApprovalPlan.detail()` → build `InfraProvisionContext` with `ProvisionPhase.APPLY` and `approvedPlan` → call `backend.provision(spec, applyContext)`
- Auto-approved nodes continue using `ProvisionPhase.APPLY` directly (current behaviour)

#### Compliance and IoT domains

Stub evaluators returning `AutoApproved` for all nodes. The evaluator is injected and the approval flow is structurally present but never triggers. Policies can be refined later without provisioner changes.

### 5. Infra type migration

| Type | From | To | Action |
|------|------|----|--------|
| `RiskClassification` | `ops.api.infra.context` | `ops.api.approval` | Move, delete original |
| `RiskThresholds` | `ops.api.infra.context` | — | Delete, replaced by `ApprovalThresholds` |
| `ProvisionPlan` | `ops.api.infra.plan` | stays | Add `implements PlanDetail` |
| `InfraProvisionContext` | `ops.api.infra.context` | stays | Field type `RiskThresholds` → `ApprovalThresholds` |

All callers update imports. No wrappers or backward-compatibility shims.

### 6. Testing approach

**Unit — shared types and handler:**
- `OpsPendingApprovalHandler`: state transitions (pending→approved→check, pending→rejected→acknowledge→removed), duplicate recordPending replaces old entry, concurrent access
- `ApprovalThresholds.requiresApproval()`: boundary conditions per risk level
- `InMemoryPlanStore`: store/retrieve/remove, missing key returns empty

**Unit — evaluators and provisioner flow:**
- `DeploymentApprovalEvaluator`: each spec type → expected risk. TrustPolicy → RequiresApproval. Channel → AutoApproved.
- `InfraApprovalEvaluator`: backend returns HIGH risk plan → RequiresApproval. Empty plan → AutoApproved.
- `DeploymentNodeProvisioner`: stub evaluator + plan store. High-risk → PendingApproval. Low-risk → Success. Re-entry with valid approval → Success. Stale approval (specHash mismatch) → re-evaluates.
- `InfraNodeProvisioner`: same structure. Re-entry verifies `InfraProvisionContext.approvedPlan` is populated from stored `ProvisionPlan`.

**Integration — full reconciliation cycle:**
Real provisioner + real evaluator + real handler + real plan store (all in-memory). Exercises:
1. Happy path: high-risk node → PendingApproval → approve → re-entry → Success
2. Auto-approve: low-risk node → Success directly
3. Rejection: PendingApproval → reject → Rejected → acknowledgeRejection → clean state
4. Stale approval: spec change between plan and approval → new PendingApproval
5. Deprovision approval: same flow through deprovision path

## Out of scope

- **`requiresHuman` + `HumanNodeHandler`**: Separate mechanism for structurally human nodes. Filed as ops#28.
- **WorkItem integration**: Creating casehub-work WorkItems on `recordPending()` and wiring completion to `approve()`. The `approve()`/`reject()` API is the integration seam — WorkItem support layers on top without changing the core handler.
- **Persistent approval storage**: `OpsPendingApprovalHandler` uses in-memory state. Persistent store is a follow-up once the mechanism is proven.
- **Responsive reconciliation**: `approve()` triggering `requestReconciliation()` on the loop. Periodic resync (5 min default) suffices initially.
- **Configurable thresholds**: Per-tenant or per-environment approval thresholds. Evaluators use static thresholds for now.
